# Чеклист перед першою міграцією — SushiChain

> **Покроковий посібник для розробника перед створенням першої PostgreSQL-міграції.**
> Проходьте кожен пункт по порядку. Усі перевірки мають бути виконані до запуску `bun run db:migrate --name init`.

---

## 1. Передумови (Prerequisites)

Переконайтеся, що інфраструктура готова до роботи з PostgreSQL.

- [ ] **Docker установлен и запущен** — Prisma потребує працюючий PostgreSQL для генерації міграції. Перевірте: `docker info`
- [ ] **docker compose -f docker-compose.dev.yml up -d выполнен успешно** — піднімає локальний PostgreSQL 16 у Docker. Перевірте: `docker compose -f docker-compose.dev.yml ps`
- [ ] **pg_isready подтверждает доступность БД** — контейнер має бути повністю готовий до прийому підключень:
  ```bash
  docker compose -f docker-compose.dev.yml exec postgres pg_isready
  ```
- [ ] **.env.development существует и содержит правильный DATABASE_URL** — цей файл використовується для локальної розробки. Перевірте, що файл існує і містить валідний рядок підключення
- [ ] **DATABASE_URL указывает на PostgreSQL (не SQLite)** — найпоширеніша помилка — залишився рядок підключення до SQLite. Перевірте, що URL починається з `postgresql://`:
  ```bash
  # Правильно:
  DATABASE_URL=postgresql://sushichain:secret@localhost:5432/sushichain_dev?schema=public
  # Неправильно:
  # DATABASE_URL=file:./dev.db
  ```
- [ ] **В проекте нет локального файла .env с SQLite DATABASE_URL (или он перезаписан)** — Bun та Prisma завантажують `.env` автоматично. Якщо в корені є `.env` з `file:./dev.db`, він може перезаписати налаштування з `.env.development`. Видаліть або виправте його

## 2. Валидация схемы (Schema Validation)

Перевірте, що канонічна схема коректна та орієнтована на PostgreSQL.

- [ ] **`bun run db:validate`** — має вивести `The schema is valid.`. Будь-яка інша відповідь означає синтаксичну помилку в `prisma/schema.prisma`
- [ ] **Файл prisma/schema.prisma содержит `provider = "postgresql"`** — це канонічна схема. Якщо провайдер досі `sqlite`, міграція згенерує неправильний SQL
- [ ] **Все модели имеют @@map("table_name")** — Prisma за замовчуванням створює таблиці з іменами моделей у PascalCase. `@@map` гарантує snake_case імені таблиць у PostgreSQL. Без цього імена будуть неконсистентними
- [ ] **Нет полей String? где должен быть Json? (для PG-совместимости)** — SQLite зберігає JSON як текст, але PostgreSQL має нативний тип `Json`. Поля, що зберігають JSON-об'єкти, мають використовувати тип `Json`, а не `String?`
- [ ] **Нет комментариев с "NOTE: For PostgreSQL"** — такі коментарі зазвичай залишаються, коли зміни з `schema.sqlite.prisma` випадково потрапили в канонічну схему. Це означає, що схема може містити SQLite-специфічні рішення

## 3. Проверка целостности схемы (Schema Consistency)

Перевірте структурну цілісність схеми перед генерацією міграції.

- [ ] **Все tenant-scoped таблицы имеют brandId** — у SushiChain використовується multi-tenant архітектура. Кожна таблиця, що належить бренду, повинна мати `brandId String` з FK на `brands.id`. Без цього дані будуть «загальними» для всіх брендів
- [ ] **Все FK-отношения используют onDelete (Cascade, SetNull, или Restrict)** — PostgreSQL вимагає явної політики видалення. Відсутність `onDelete` може призвести до помилок при видаленні записів. Виберіть:
  - `Cascade` — видалити дочірні записи разом з батьківським
  - `SetNull` — встановити FK в `null` при видаленні
  - `Restrict` — заборонити видалення, якщо є дочірні записи
- [ ] **Нет циклических зависимостей между моделями** — циклічні FK-залежності ускладнюють міграцію і можуть призвести до зависань при видаленні. Перевірте граф залежностей
- [ ] **Enum-типы не содержат зарезервированных PG слов** — PostgreSQL має зарезервовані слова (`user`, `order`, `group`, `table` тощо). Якщо enum-значення або імена таблиць використовують такі слова, потрібні екранування `"order"` або перейменування
- [ ] **Все @@unique ограничения корректны** — перевірте, що унікальні обмеження не суперечать бажаній логіці. Наприклад, `@@unique([brandId, slug])` замість просто `@@unique([slug])` для multi-tenant схеми
- [ ] **Все @@index покрытия достаточны для основных запросов** — кожна модель, що фільтрується по `brandId`, повинна мати `@@index([brandId])`. Перевірте покриття індексів для основних query-шляхів

## 7. Проверка кода на PG-совместимость

Детальна перевірка коду на сумісність з PostgreSQL.

- [ ] **Нет `raw()` запросов с SQLite-специфичным синтаксисом** — SQLite підтримує нестандартний синтаксис, який не працює в PostgreSQL. Шукайте:
  - `datetime('now')` → замінити на `NOW()`
  - `AUTOINCREMENT` → замінити на `SERIAL` / `GENERATED ALWAYS AS IDENTITY`
  - Параметри прив'язки `?` → замінити на `$1, $2, ...`
- [ ] **Нет использования `sqlite3` модуля напрямую** — прямі імпорти `sqlite3` або `better-sqlite3` не працюватимуть з PostgreSQL. Всі запити мають йти через Prisma Client
- [ ] **Json-поля обрабатываются корректно (не как строки)** — у SQLite JSON зберігається як текст і потребує `JSON.parse()` / `JSON.stringify()`. У PostgreSQL Prisma автоматично серіалізує/десеріалізує `Json` поля. Приберіть ручну серіалізацію
- [ ] **String[] поля (tags, allergens) используются через Prisma, не через raw SQL** — масиви в PostgreSQL реалізовані через нативні типи, а не через JSON. Використовуйте Prisma API для роботи з такими полями
- [ ] **Нет LIMIT/OFFSET без ORDER BY (в PG это не детерминировано)** — у PostgreSQL без `ORDER BY` порядок рядків не гарантується. `LIMIT 10` без сортування може повертати різні результати при кожному виклику

## 8. Проверка seed-данных

Перевірте, що seed-дані сумісні з PostgreSQL-схемою.

- [ ] **prisma/seed.ts использует модели, совместимые с PG-схемой** — всі моделі, що використовуються в seed, мають відповідати моделям з `prisma/schema.prisma`, а не `schema.sqlite.prisma`
- [ ] **Seed не зависит от SQLite-специфичных особенностей** — наприклад, відсутні `JSON.stringify()` для Json-полів (Prisma для PostgreSQL приймає plain objects), відсутні SQLite-специфічні типи даних
- [ ] **Seed создает данные для всех брендов и tenant-объектов** — переконайтеся, що seed створює дані для кожного бренду (multi-tenant), а не тільки для одного

## 9. SQL Review (после генерации миграции)

Після виконання `bun run db:migrate --name init` ретельно перевірте згенерований SQL.

```bash
cat prisma/migrations/$(ls -t prisma/migrations/ | head -1)/migration.sql
```

- [ ] **Миграция содержит CREATE TYPE для всех 10 enum-ов** — PostgreSQL створює enum-типи через `CREATE TYPE`. Перевірте наявність: `OrderType`, `OrderStatus`, `PaymentStatus`, `PaymentMethod`, `PromotionType`, `PromotionStatus`, `LoyaltyTransactionType`, `FeedbackType`, `FeedbackStatus`, `CampaignType`, `CampaignStatus`
- [ ] **Миграция содержит CREATE TABLE для всех 27 таблиц** — порахуйте `CREATE TABLE` в SQL. Має бути 27 таблиць: `brands`, `users`, `user_sessions`, `user_addresses`, `branches`, `delivery_zones`, `categories`, `products`, `product_option_groups`, `product_options`, `favorite_products`, `carts`, `cart_items`, `orders`, `order_items`, `payments`, `promotions`, `loyalty_accounts`, `loyalty_transactions`, `reviews`, `feedbacks`, `campaigns`, `campaign_messages`, `couriers`, `delivery_assignments`, `menu_translations`, `brand_settings`
- [ ] **Миграция содержит CREATE INDEX для всех индексов** — кожен `@@index` та `@unique` у схемі має відповідний `CREATE INDEX` або `UNIQUE` обмеження в SQL
- [ ] **Нет DROP TABLE, DROP COLUMN, ALTER TYPE (в init-миграции их быть не должно)** — це init-міграція, яка створює БД з нуля на порожній базі. Будь-які `DROP` означають, що Prisma виявив розбіжність і намагається щось видалити — це ознака проблеми
- [ ] **Все FK-ограничения корректны** — перевірте, що кожен `REFERENCES` вказує на правильну таблицю та колонку, а `ON DELETE` політика відповідає схемі
- [ ] **Порядок таблиц учитывает зависимости FK** — таблиці, на які посилаються FK, мають створюватись раніше, ніж таблиці, що посилаються на них. Prisma зазвичай	handle це автоматично, але варто перевірити

## 10. Опасные операции (Dangerous Operations Check)

Фінальна перевірка згенерованого SQL на небезпечні операції.

- [ ] **В миграции нет DROP TABLE** — незворотна втрата даних. В init-міграції їх бути не повинно взагалі
- [ ] **В миграции нет DROP COLUMN** — незворотна втрата даних. Якщо Prisma згенерував `DROP COLUMN` — схема, швидше за все, не відповідає очікуванням
- [ ] **В миграции нет ALTER TYPE ... RENAME (в init не нужно)** — перейменування enum-значень в init-міграції означає, що enum вже існує, що неможливо для першої міграції на порожній БД
- [ ] **В миграции нет ADD COLUMN ... NOT NULL без DEFAULT** — на порожній БД це безпечно, але це поганий патерн. Для майбутніх міграцій (на таблицях з даними) такий SQL впаде
- [ ] **В миграции нет данных (INSERT/UPDATE) — только DDL** — init-міграція має містити тільки `CREATE TABLE`, `CREATE TYPE`, `CREATE INDEX`. Будь-які `INSERT` / `UPDATE` означають, що в SQL потрапили дані, яких там бути не повинно

## 11. Git-готовность

Підготовка до коміту міграції.

- [ ] **prisma/schema.prisma и prisma/migrations/ будут закоммичены ОДНИМ коммитом** — схема та міграція мають бути нерозривно пов'язані в Git-історії. Розрив між ними призведе до помилок у інших розробників:
  ```bash
  git add prisma/schema.prisma prisma/migrations/
  git commit -m "feat(db): init migration"
  ```
- [ ] **Коммит-сообщение следует формату: "feat(db): init migration"** — використовуйте conventional commits. Формат: `<type>(<scope>): <опис>`. Для міграцій: `feat(db):`, `fix(db):`
- [ ] **schema.sqlite.prisma обновлён (если были изменения в PG-схеме)** — якщо ви вносили зміни в канонічну схему, перенесіть відповідні зміни в SQLite-копію для локального прототипування. Це не обов'язково, але корисно для консистентності
